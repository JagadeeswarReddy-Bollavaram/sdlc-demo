# Skill: detect-stack

## Purpose
Analyse the current repository and produce `.claude/stack-manifest.yml` — the single
source of truth consumed by `generate-pipeline` and `deploy`.

There is no IaC generation, no infra provisioning, and no compute target selection.
All applications deploy as pods in the shared AKS cluster. The only things this skill
determines are: runtime, version, database type, and ADO/ACR references from config.yml.

## Trigger
- `cc run detect-stack`
- Auto-invoked by `deploy` and `generate-pipeline`
- Auto-invoked by pre-commit hook when project files change

---

## Detection Logic

### 1. Runtime & Version

| File / Pattern                        | Runtime  | Version source                        |
|---------------------------------------|----------|---------------------------------------|
| `*.csproj`, `*.sln`                   | `dotnet` | `<TargetFramework>net8.0` → `8.0`     |
| `package.json`                        | `node`   | `engines.node`, else latest LTS       |
| `requirements.txt` / `pyproject.toml` | `python` | `.python-version` or `python_requires`|
| `pom.xml` / `build.gradle`            | `java`   | `<java.version>` in pom.xml           |
| `go.mod`                              | `go`     | `go` directive in go.mod              |

### 2. Framework Detection (informational — used in pipeline displayName only)

| Runtime | Signal                              | Framework      |
|---------|-------------------------------------|----------------|
| dotnet  | `AzureFunctionsVersion` in .csproj  | `functions`    |
| dotnet  | `<Project Sdk="Microsoft.NET.Sdk.Web">` | `aspnetcore` |
| dotnet  | `<Project Sdk="Microsoft.NET.Sdk">` | `minimal-api`  |
| node    | `react` / `next` in dependencies    | `react` / `next` |
| node    | `express` / `@nestjs/core`          | `express` / `nestjs` |
| python  | `fastapi` / `django` / `flask`      | matching name  |

### 3. Build File Detection (dotnet only)

| Signal                            | Manifest field           |
|-----------------------------------|--------------------------|
| Single `*.sln` at root            | `build.solution_file`    |
| No `.sln`, single `*.csproj`      | `build.solution_file` = that .csproj |
| `*Tests*.csproj` present          | `build.test_project`     |
| `coverlet.runsettings` present    | `build.test_runsettings` |

### 4. Database Detection (informational — no IaC generated)

| Signal                                        | `database.type`  |
|-----------------------------------------------|------------------|
| EF `UseSqlServer` / SQL conn string           | `azure_sql`      |
| EF `UseNpgsql` / `psycopg2` in requirements  | `postgresql`     |
| EF `UseCosmos` / `pymongo` / `MONGO_URI`      | `cosmosdb`       |
| None of the above                             | `none`           |

### 5. Kubernetes Manifests Detection

Check whether a `k8s/` directory already exists in the repo:
- `k8s/` present → `k8s_manifests: existing` — pipeline will use them as-is
- `k8s/` absent  → `k8s_manifests: generate` — `generate-pipeline` will create starter manifests

## Output — `.claude/stack-manifest.yml`

```yaml
# Auto-generated by detect-stack. Do not edit manually.
generated_at: "2024-01-15T10:30:00Z"
schema_version: "3.0"

# ── Application identity ──────────────────────────────────────────────────
app_name: "{{REPO_NAME}}"       # derived from git remote repo name, lowercase-hyphenated
ado_repo: "{{REPO_NAME}}"       # Azure Repos repo name within the shared ADO project

# ── Runtime ───────────────────────────────────────────────────────────────
runtime:   dotnet               # dotnet | node | python | java | go
framework: aspnetcore           # informational — used in pipeline display names
version:   "8.0"                # SDK/runtime version string

# ── Build (dotnet) ────────────────────────────────────────────────────────
build:
  solution_file:   "MyApp.sln"
  test_project:    "MyApp.Tests/MyApp.Tests.csproj"
  test_runsettings: "MyApp.Tests/coverlet.runsettings"

# ── Kubernetes ────────────────────────────────────────────────────────────
# All apps deploy as pods in the shared AKS cluster.
# Namespace convention: {{APP_NAME}}-dev | {{APP_NAME}}-staging | {{APP_NAME}}-production
# Namespaces are created by the CD pipeline on first deploy if they don't exist.
k8s_manifests: generate         # generate | existing
                                # generate → generate-pipeline creates starter k8s/ files
                                # existing → pipeline uses k8s/ files already in repo

# ── Database (informational — no IaC generated) ───────────────────────────
database:
  type: azure_sql               # azure_sql | postgresql | cosmosdb | none
  # Connection strings are managed in Key Vault by the platform team.
  # Reference them in k8s/configmap.yml or as CSI driver secrets.

# ── CI/CD ─────────────────────────────────────────────────────────────────
ci_cd:
  ado_project:        "{{ADO_PROJECT}}"         # from config.yml
  ado_repo:           "{{REPO_NAME}}"
  service_connection: "{{SERVICE_CONNECTION}}"  # from config.yml — shared across all apps

# ── Versioning ────────────────────────────────────────────────────────────
versioning:
  major: "1"
  minor: "0"

previous_manifest_hash: ""
```

---

## Diff Behaviour

1. Compute SHA256 of new manifest (exclude `generated_at`, `previous_manifest_hash`)
2. Compare against stored hash
3. If changed: print diff → set `STACK_CHANGED=true` → prompt "Regenerate pipeline? [Y/n]"
4. If unchanged: print "Stack unchanged. No action needed."

## Dockerfile Detection & Warning

After writing the manifest, explicitly check for a Dockerfile:

```bash
if [ ! -f "Dockerfile" ]; then
  echo ""
  echo "##vso[task.logissue type=warning]⚠️  No Dockerfile found."
  echo "   containerized: false has been written to stack-manifest.yml."
  echo "   The CI pipeline Stage 4 (Containerise) will fail without one."
  echo ""
  echo "   Run:  cc run generate-dockerfile"
  echo "   This will generate a production-ready multi-stage Dockerfile"
  echo "   for your runtime ({{RUNTIME}} {{RUNTIME_VERSION}})."
  echo ""
  echo "   Or if deploy --init is running, it will call generate-dockerfile"
  echo "   automatically before proceeding."
fi
```

Set `containerized: false` in the manifest when Dockerfile is absent.
Set `containerized: true` when Dockerfile is present.
